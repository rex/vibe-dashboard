# Live agent & workflow monitoring — design

How Vibe should watch the LIVE output of coding agents (plain Claude Code subagents AND full workflow agents) in real time, and track a workflow as it advances — including the hard requirement: **when a workflow agent finishes a step and moves to the next phase, the live view must pick that up.** Grounded in the actual on-disk shapes (verified 2026-07-02). Companion to [BACKLOG.md](../BACKLOG.md).

## Current state

Live agents are detected by `Shared/Scan/AgentProbe.swift`: one `ps -axo pid,etime,command` snapshot, filter lines containing `claude`/`codex`, resolve each PID's cwd via `lsof -d cwd`, and map cwd onto the deepest matching repo. `FleetScanner` (lines ~73-80) turns each hit into a coarse `AgentInfo(active:true, tool, elapsed, note:"live edit session")` stored on `Repo.agent`. The UI reads it as a boolean dot (`Fleet.agentsActive`, `WorkspaceDetailView.agentsActive`, an `AgentsView.swift` screen). Refresh is PULL-only: `FleetStore.rescan()` runs the whole fleet scan on launch (`.task`) and on manual triggers (toolbar/menu/⌘K) — there is NO timer, NO FSEvents, NO streaming. Limits: (1) it is a boolean, carrying no live output; (2) a workflow's 11 parallel subagents share ONE parent PID and ONE cwd, so ps/lsof collapses them to a single dot — it literally cannot see individual agents or count them; (3) it has NO concept of phase; (4) it misses headless/detached sessions; (5) state is stale between manual rescans. It satisfies "a repo has an agent in it" and nothing more.

## Live data sources on disk

- MAIN TRANSCRIPT — ~/.claude/projects/<path-slug>/<sessionId>.jsonl. One JSON event per line, APPENDED live. Line keys: type (assistant|user|last-prompt|mode|attachment|queue-operation|custom-title), message{role,content[]}, uuid, parentUuid, timestamp, cwd, sessionId, gitBranch, version. tool_use blocks live in message.content[] with {type:'tool_use',name,input}; results come back as type:'user' lines with tool_result blocks. The orchestrator's own Task/Agent spawns appear here as tool_use name='Agent' {description,subagent_type,model,prompt}. NOTE: no structured workflow_phase/workflow_agent events are emitted here live — only the raw script text echoed inside an assistant message.
- PLAIN SUBAGENT TRANSCRIPTS — ~/.claude/projects/<slug>/subagents/agent-<id>.jsonl  + a sibling agent-<id>.meta.json = {agentType, description, toolUseId}. These are Task-tool subagents. isSidechain=true. Append live.
- WORKFLOW AGENT TRANSCRIPTS (the live phase source) — ~/.claude/projects/<slug>/subagents/workflows/wf_<runId>/agent-<id>.jsonl + agent-<id>.meta.json = {agentType:'workflow-subagent', spawnDepth}. The wf_<runId> DIRECTORY groups all agents of one workflow run. Each agent JSONL's first line carries cwd, gitBranch, slug (e.g. 'adaptive-honking-bonbon'), and the fan-out prompt as the first user message. VERIFIED LIVE: run wf_4c5e2d38-a3e had 11 agent JSONLs appending in real time during this investigation.
- COMPLETED WORKFLOW SUMMARY (history/enrichment only, NOT live) — ~/.claude/projects/<slug>/workflows/wf_<runId>.json, written at completion. Keys: runId, taskId, status(completed|killed|failed), startTime, durationMs, agentCount, workflowName, phases:[{title,detail}], and workflowProgress:[ {type:'workflow_phase',index,title}, {type:'workflow_agent',index,label,phaseIndex,phaseTitle,agentId,model,state(queued|progress|done|error),startedAt,lastToolName,lastProgressAt,tokens,toolCalls,durationMs,promptPreview,resultPreview} ]. Plus workflows/scripts/<name>-wf_<id>.js (the phase()/agent() script) — ALSO only written at completion. Across 90 runs: states done=1021,error=202,progress=8.
- AGENTSVIEW DB (supplementary tree + history) — ~/.agentsview/sessions.db (WAL, ~2GB), read via /usr/bin/sqlite3 -readonly -json (pattern already in AgentsViewProbe.swift). sessions row per agent incl. workflow subagents (relationship_type='subagent', parent_session_id=orchestrator, cwd, message_count climbing live, outcome='unknown' until done, file_mtime, health_grade). tool_calls(tool_name,skill_name,subagent_session_id,category). VERIFIED it indexes the live workflow children — but there is NO phase/workflow table, so phases are invisible to it.

## Approaches considered

### A. ps/lsof snapshot only (status quo, extended)
**How:** Keep polling `ps -axo` + `lsof` each rescan, add a fast 2-3s cadence, map cwd→repo. This is what `AgentProbe.sessions()` does now.

**Trade-offs:** Zero new file parsing, already Sendable-clean. But it can ONLY ever answer 'is a claude/codex process alive in this cwd' — a boolean. It sees NO output, NO subagents (all 11 workflow children share ONE parent PID with the SAME cwd, so lsof collapses them to one dot), and has NO notion of phase. It also misses headless/detached agents. Fundamentally cannot satisfy the phase-transition requirement. Keep it only as a cheap liveness cross-check.

### B. Tail the per-agent JSONL transcripts (RECOMMENDED core)
**How:** Watch `~/.claude/projects/<slug>/**` with one FSEvents stream. Parse the main `<session>.jsonl` and every `subagents/**/agent-*.jsonl` incrementally (seek to last byte offset, read appended lines only). Each line gives cwd, timestamp, and message.content tool_use/tool_result/text — the real live output. Group workflow-agent transcripts by their parent `wf_<id>` directory.

**Trade-offs:** This is the ONLY source that updates truly live for BOTH plain subagents and workflow agents (verified: agent JSONLs append in real time — this very subagent's file grew from 24→49 lines mid-investigation). Cost is a JSONL parser + byte-offset bookkeeping. Phase must be inferred (see phaseTracking) because the phase LABEL is not in the transcript lines — it only lands in the terminal `wf_*.json`. Files must stay <400 lines, so split parser/watcher/model.

### C. Tail the completed wf_*.json workflowProgress
**How:** Read `workflows/wf_<id>.json` which has a rich `workflowProgress` array (workflow_phase + workflow_agent records with phaseIndex, phaseTitle, state, lastToolName, tokens, toolCalls).

**Trade-offs:** This is the PERFECT schema for phase tracking — but it is written ONLY at workflow completion (status: completed/killed/failed). Verified: the currently-running `wf_4c5e2d38-a3e` has its 11 agent transcripts on disk but NO wf_*.json and NO script file yet. So it is useless for LIVE monitoring; it is excellent for post-hoc/history. Use it to enrich finished runs and to learn the phase list, not for the live view.

### D. Poll AgentsView sessions.db
**How:** Reuse the existing `sqlite3 -readonly` CLI pattern (AgentsViewProbe) to poll `sessions`/`tool_calls`, joining subagents to parents via `relationship_type='subagent'` + `parent_session_id`.

**Trade-offs:** Verified it DOES index live workflow subagents (rows appear with climbing message_count) and gives a clean parent→child tree + skill_name + cwd with no JSONL parsing. BUT: it has NO workflow/phase table (phases are invisible to it), `outcome` stays 'unknown' until done, and it lags the raw files. Good as a supplementary tree/liveness source and history backstop; cannot deliver phase transitions. Degrade-gracefully like the existing probe.

## Recommendation

Add a dedicated LIVE probe/service that TAILS the per-agent JSONL transcripts (approach B) as the source of truth, driven by ONE FSEvents stream over ~/.claude/projects, with a bounded fallback poll; enrich (not drive) with the terminal wf_*.json (C) and the AgentsView DB (D); demote ps/lsof (A) to a cheap liveness cross-check. Concretely: a new `Shared/Scan/` layer that (1) discovers active sessions = project-slug dirs whose main jsonl OR any subagents/**/agent-*.jsonl has mtime within a freshness window (e.g. last 90s) and whose owning `claude` PID is alive; (2) for each, incrementally tails from a stored byte offset, parsing only appended lines into a compact live model; (3) groups workflow agents by their `wf_<runId>` dir and runs the phase-inference state machine above; (4) publishes into a NEW `@MainActor @Observable LiveAgentStore` (SEPARATE from FleetStore so the heavy full-fleet scan and the light live tail have independent cadences and the live view never blocks on a rescan). Wire it so a live session still cross-links to a Repo by cwd (reuse FleetScanner's deepest-prefix match) to light the existing repo dot, but the rich live data lives in LiveAgentStore. This respects every constraint: un-sandboxed file reads + the existing `sqlite3 -readonly` and ProcessRunner patterns; strictly off-main-actor parsing feeding a MainActor store; and NO CPU runaway — FSEvents (event-driven, idle=0 CPU) with a coalescing latency (~0.3-0.5s) plus a slow safety poll (2-3s ONLY while ≥1 live session exists, otherwise idle), never a repeatForever/TimelineView ticker.

## Phase-transition detection (the hard part)

THE HARD PART, grounded in what's on disk. The phase LABEL a workflow assigns via `agent(prompt,{phase:'Author'})` is NOT present in any live transcript line — it is persisted only in the terminal `wf_<id>.json`'s workflowProgress, which does not exist until the run ends. So live phase detection cannot read a phase field; it must be INFERRED from the wf_<runId>/ directory's structure and file lifecycle, which ARE live. Model a workflow run as its `subagents/workflows/wf_<runId>/` directory. A PHASE = a cohort (generation) of agent-*.jsonl files that appear together. Detection algorithm, all from FSEvents-driven cheap stat/tail:  (1) COHORT BY BIRTH TIME — cluster the run's agent JSONLs by first-line timestamp (or file btime). Agents born within a small window (~a few seconds) are one phase's fan-out; a later cluster that appears only AFTER the earlier cohort's files stop growing is the NEXT phase. (2) PHASE ADVANCE SIGNAL — the transition N→N+1 fires when: every agent in the current cohort has gone quiescent (its JSONL's last line is a terminal assistant message / StructuredOutput tool_use and its mtime has been static > debounce, ~1.5-2s) AND a brand-new agent-*.jsonl is created in the same wf_ dir. New-file-creation in the wf_ dir is the crisp, real-time edge FSEvents delivers — that is the moment to emit "phase advanced". (3) PER-AGENT STEP within a phase — tail each agent JSONL: last tool_use name = current step (lastToolName), a StructuredOutput/return-shaped tool_use or the file going static = that agent done. Agent state machine: queued(file exists, 0 assistant lines) → running(appending) → done(static + terminal) / error(tool_result error tail). (4) PHASE LABELS — live view shows ordinal phases ("Phase 1 · 11 agents", "Phase 2 · 3 agents"); the moment the run finishes and wf_<id>.json lands, backfill the real phaseTitle strings from workflowProgress and reconcile. (5) SEQUENTIAL vs PARALLEL both covered: sequential steps = successive single-agent cohorts (cohort of 1, next appears after prev quiesces); parallel fan-out = a multi-file cohort. (6) Cross-check with AgentsView parent_session_id and, if present, the live orchestrator main transcript's Agent tool_use spawns to confirm the run's agent roster. Net: file-birth clustering + quiescence + new-file-in-dir gives live N→N+1 without any phase field, and the terminal json upgrades ordinals to titles.

## Implementation sketch

FILES (each well under 400 lines; split by concern):

1) Shared/Scan/TranscriptTailer.swift — actor. Maintains [path: UInt64 offset]. `func newLines(_ path) -> [Data]`: open, seek(toOffset:), read to EOF, split on \
, advance offset (handle truncation/rotation by resetting offset if file shrank). Pure IO, Sendable-clean, off main actor.

2) Shared/Scan/TranscriptEvent.swift — decode one JSONL line into `struct LiveEvent: Sendable { ts: Date; kind: .text/.toolUse(name)/.toolResult(isError)/.spawn; role; }`. Small Codable over the subset of keys (type, timestamp, message.content[]).

3) Shared/Scan/LiveAgentProbe.swift — discovery + assembly. Enumerate ~/.claude/projects/*/ ; find fresh main + subagents/**/agent-*.jsonl (mtime within window). Read each agent's meta.json (agentType: workflow-subagent | general-purpose) + first line (cwd, slug). Group workflow agents by parent `wf_<runId>` dir. Emit the model below. Reuses ProcessRunner only for the liveness PID cross-check + AgentsView parent map.

4) Shared/Scan/WorkflowPhaseTracker.swift — the state machine: ingest each run's agent set, cluster by first-line ts into phases (ordinals), compute per-agent state (queued/running/done/error) from tail + mtime quiescence, and detect N→N+1 (all current-cohort quiescent + new agent-*.jsonl born). Emits `phaseAdvanced` deltas.

5) Shared/Models/LiveAgent.swift — Sendable value types: `LiveSession{ id; kind:.plain/.workflow; cwd; repoId?; tool; startedAt; agents:[LiveAgentRun] }`, `LiveAgentRun{ agentId; label?; phaseIndex; phaseTitle?; state; lastToolName?; lastText?; tokens?; toolCalls; lastActivity }`, `WorkflowRun{ runId; phases:[LivePhase]; currentPhaseIndex }`.

6) Shared/Services/LiveAgentStore.swift — @MainActor @Observable. Holds `sessions:[LiveSession]`. `start()` opens the FSEvents stream; on coalesced events, hops to the tailer actor, diffs, republishes. `stop()` on view disappear. Independent of FleetStore.

7) Shared/Scan/ClaudeProjectsWatcher.swift — thin FSEventStreamCreate wrapper (kFSEventStreamCreateFlagFileEvents, latency ~0.4s) over ~/.claude/projects, callback → async channel. Idle when no events. Fallback: an async loop that sleeps 2-3s ONLY while sessions non-empty (Task.sleep, cancellable) — never repeatForever.

8) VibeDashboard/Views/LiveAgentsView.swift (replace/augment AgentsView.swift) — a real-time view: one card per live session; for workflows, a phase rail (Phase 1 ▸ Phase 2 …) with the current phase lit acid-lime, each agent a row showing state + lastToolName + a tail of latest output text; "phase advanced" animates the rail. Reads LiveAgentStore only.

ENRICHMENT: on wf_<id>.json appearance (FSEvents create in workflows/), read workflowProgress once to upgrade ordinal phases → real titles and finalize states; keep AgentsViewProbe-style read-only sqlite for the parent→child roster + history when transcripts have rotated. Keep AgentProbe.sessions() as a liveness sanity cross-check only. CADENCE: FSEvents-driven (idle 0 CPU) + optional slow poll gated on sessions.isEmpty==false; MainActor only republishes diffs — no per-row timers, no TimelineView(.animation), honoring the idle-CPU rule.

## Risks & caveats

(1) Phase LABEL is unavailable live — only ordinals until the terminal wf_*.json lands; mitigated by ordinal-then-backfill, but a workflow with heterogeneous sequential single-agent steps could be mis-clustered if two phases' agents are born within the debounce window — tune the quiescence window (~1.5-2s static mtime) and require a new-file edge, accepting rare 1-cohort merges over false phase churn (anti-alert-fatigue: never flash a phantom phase). (2) FSEvents on ~/.claude/projects can be chatty (2GB WAL, many files) — must coalesce and filter to *.jsonl/*.json under subagents|workflows, and store byte offsets to avoid re-parsing whole files (some agent JSONLs are 600KB+). (3) Truncation/rotation: a shrunk file must reset the stored offset or the tailer desyncs. (4) 'Live' freshness by mtime can false-positive on a just-finished run; gate with PID-alive cross-check + a short freshness window. (5) Slug/path-slug decoding: the project dir name is a lossy slug of cwd (e.g. -Users-dev-Code---APPS-...) — map via the first transcript line's real `cwd`, never by de-slugging the dir name. (6) Strict Swift 6 concurrency: FSEvents callback is a C context — bridge through an @unchecked Sendable box + async channel like ProcessRunner's IOBox; keep all mutable tail state inside an actor. (7) Keep the live tail OFF the full-fleet rescan path so a slow git scan never stalls the live view (that priority-inversion class already bit this app once). (8) Files >400 lines: the split above keeps each concern small; watch WorkflowPhaseTracker.
